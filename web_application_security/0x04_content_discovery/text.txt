The reason it feels like you're watching paint dry is that TFTP (Trivial File Transfer Protocol) is a relic from 1981. It wasn't built for speed or the modern internet; it was built to fit into the tiny memory of 1980s computer chips.

Here is the technical breakdown of why this specific "attack" (brute-forcing) is so slow:

1. The "Stop-and-Wait" Mechanism
Unlike HTTP, which sends a huge stream of data all at once, TFTP is synchronous.

The client sends a request.

The client must wait for the server to say "Okay, I'm here" (an ACK) before it can do anything else.

If you are brute-forcing, Gobuster has to wait for that "Yes" or "No" for every single word in your list before moving to the next.

2. UDP is "Blind" and "Unreliable"
TFTP uses UDP instead of TCP.

TCP is like a phone call; you know immediately if the other person hung up.

UDP is like sending a postcard; you have no idea if it arrived until you get a response back.

Because of this, Gobuster has to wait the full timeout period (e.g., 5 seconds) for every "wrong" guess just to be sure the server isn't just being slow.

3. Server-Side Bottlenecks
TFTP servers are usually single-threaded and very lightweight. If you try to run 10 or 20 threads (like we do with Web discovery), the TFTP server often gets "confused" or overwhelmed and starts dropping packets. This leads to the "Transfer timed out" errors you saw earlier.
